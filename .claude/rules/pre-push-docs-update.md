# ルール: push前のドキュメント・メモリ更新チェック

## トリガー条件
`git push` を実行する前に、必ずドキュメントやメモリの更新が必要かチェックする。

## ルール
コード変更を含むブランチをリモートにpushする前に、以下のドキュメント群を確認し、必要に応じて更新・コミットすること。

### 1. チェック対象

| 対象 | パス | 確認観点 |
|------|------|----------|
| README.md | `README.md` | 機能一覧・プロジェクト構成・確認手順が実装と一致しているか |
| CLAUDE.md | `CLAUDE.md` | プロジェクト構成・開発コマンド・注意事項が最新か |
| プロダクト仕様書 | `docs/PRODUCT_SPEC.md` | 機能一覧・デザイン方針・技術スタックが実装と一致しているか |
| ルール | `.claude/rules/` | 新しいルールの追加や既存ルールの修正が必要か |
| メモリ | auto memory | プロジェクト状況・ユーザー情報の更新が必要か |
| Docs | `docs/` など | 技術ドキュメントが存在する場合、内容が最新か |

### 2. チェック手順

1. `git diff` または `git log` で今回の変更内容を把握する
2. 上記の各ドキュメントを読み、変更内容との乖離がないか確認する
3. 乖離がある場合は修正してコミットする
4. 重要でない軽微な差異（表現の好みレベル）はスキップしてよい

### 3. 更新が必要な典型的ケース

- **新規ファイル追加**: README.md / CLAUDE.md のプロジェクト構成に反映
- **新機能追加**: README.md の機能一覧・確認手順に反映
- **API・データモデル変更**: CLAUDE.md の注意事項、プロダクト仕様書の技術スタックに反映
- **開発フロー変更**: CLAUDE.md の開発コマンド、ルールの追加・修正
- **重要な設計判断**: メモリへの記録（将来の会話で参照できるように）

### 4. 更新時のコミットルール

- ドキュメント更新は本体のコード変更とは別コミットにする
- コミットメッセージは `docs:` プレフィックスを使用する

## 対象外
- `.swift` ファイルの変更を含まないpush（ドキュメントのみの修正など）はチェック不要
