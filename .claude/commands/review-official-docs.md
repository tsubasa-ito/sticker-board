# 公式ドキュメント準拠レビュー

変更されたコードを、使用している各技術の**公式ドキュメント・推奨パターン**に基づいてレビューする。
プロジェクト固有ルール（.claude/rules/）の遵守チェックではなく、言語・ライブラリの公式が推奨する実装・設計になっているかを検証する。

## 手順

### 1. 変更差分の取得
`git diff HEAD` および `git diff --cached` を実行し、変更されたSwiftファイルを特定する。
未コミットの変更がない場合は `git diff HEAD~1 HEAD` で直前のコミットの差分を取得する。

### 2. 使用技術の特定
変更されたコードから、以下の技術の使用箇所を特定する：
- **Swift** (言語仕様・コーディング規約)
- **SwiftUI** (ビュー構成・状態管理・ライフサイクル)
- **SwiftData** (モデル定義・クエリ・リレーション)
- **Vision Framework** (画像処理・リクエスト実行)
- **Combine / async-await** (非同期処理パターン)
- **Foundation** (データ型・ファイル操作)

### 3. 公式ドキュメントの参照
context7 MCP を使用して、特定した各技術の最新公式ドキュメントを取得する。

以下の手順で実施する：
1. `resolve-library-id` で各技術のライブラリIDを解決
2. `query-docs` で該当する公式ドキュメントを取得
3. 変更コードと公式推奨パターンを照合

### 4. レビュー観点
各技術について以下の観点でレビューする：

#### Swift
- Swift API Design Guidelines への準拠（命名規約、引数ラベル）
- 値型 vs 参照型の適切な使い分け
- Optional の安全な扱い（force unwrap の回避）
- アクセス制御の適切な設定

#### SwiftUI
- View の責務分離（大きすぎるViewの検出）
- @State / @Binding / @Environment の適切な使い分け
- body の計算コスト（重い処理が含まれていないか）
- .task / .onAppear の使い分け
- List / LazyVStack の適切な選択

#### SwiftData
- @Model マクロの正しい使用
- @Query の適切なフィルタリング・ソート
- ModelContext の適切なスコープ管理
- リレーションの削除ルール設定

#### Vision Framework
- VNRequest のライフサイクル管理
- リクエスト実行のスレッド管理
- エラーハンドリングパターン

#### async/await
- MainActor の適切な使用
- Task のキャンセル対応
- 構造化並行性の活用

### 5. レポート出力
以下のフォーマットで結果を出力する：

```
## 公式ドキュメント準拠レビュー結果

### 総合評価: [A / B / C / D]
- A: 公式推奨パターンに完全準拠
- B: 軽微な改善点あり（動作に影響なし）
- C: 推奨パターンからの逸脱あり（改善推奨）
- D: 重大な問題あり（修正必須）

### 検出事項

#### [技術名] - [重要度: 高/中/低]
- **該当箇所**: `ファイル名:行番号`
- **現在の実装**: （コード抜粋）
- **公式推奨**: （ドキュメントからの引用・リンク）
- **推奨修正**: （具体的な修正案）

### 準拠確認済み
- （問題なしと確認した項目のリスト）
```

### 6. 注意事項
- プロジェクト固有の設計判断（CLAUDE.md記載事項）は尊重し、指摘対象外とする
- deprecatedなAPIの使用は必ず指摘する
- iOS 18+ をターゲットとしているため、それ以前のworkaroundは不要と判断する
